0. Contexte, périmètre et principes directeurs

La Card Payment Data Platform (CPDP, « socle data ») centralise les données de l'acquisition monétique — transactions, autorisations, captures, remises, chargebacks, fraude — produites par des systèmes hétérogènes (NEPTING, K2/Partecis via BCEF-IT/DHF, E-Stream, Computop, fichiers batch IBM COS) et les met à disposition des consommateurs : Accepta Vision Portal, plateforme de reporting Axepta, extractions réglementaires et comptables.

La présente étude pose les fondations du socle sur deux volets : l'ingestion (contrats, pipeline raw vers Iceberg, pipeline transactionnelle vers MongoDB, corrélation) et le modèle organisé (indépendance sémantique, trajectoire v1 → v2). Le volet exposition et extraction de rapports volumineux est traité dans une étude dédiée.

Principe fondateur — deux représentations, une seule vérité

La même donnée existe dans deux représentations physiques, alimentées en parallèle depuis Kafka et optimisées pour des usages opposés :

  • Le Raw Lake Iceberg/Parquet : capture immuable et exhaustive des événements tels que reçus. Source de vérité éternelle, rejouable, jamais modifiée. Socle des analyses, des reconstructions et des migrations.
  • Le modèle organisé MongoDB : représentation structurée, corrélée, dans le langage métier de l'acquisition. Optimisée pour la lecture chaude et l'exposition.

Chaque zone est alimentée par projection depuis le flux, jamais par copie latérale entre zones servies. Ce découplage est ce qui rendra la migration v1 → v2 du modèle organisé possible sans drame (cf. §2.4).

1. Ingestion

1.1 Le contrat d'ingestion — et sa distinction avec le modèle organisé

Le contrat d'ingestion est un accord formel, versionné et testable entre chaque producteur et la plateforme. Il n'est pas une page de documentation : c'est un artefact de code (schémas Avro dans un Schema Registry, règles de qualité exécutables) dont la violation est détectée automatiquement, à la frontière, avant toute ingestion.

Ce que le contrat spécifie

DimensionContenu contractuel
SchémaSchéma Avro versionné dans le Schema Registry (un subject par topic). Format de transport : Avro binaire (recommandé) ou JSON validé contre le schéma. Politique d'évolution : BACKWARD_TRANSITIVE — un producteur ne peut jamais publier un changement qui casserait les consommateurs existants.
Identitésevent_id globalement unique (idempotence), event_type (AUTHORIZATION, CAPTURE, REMITTANCE, CHARGEBACK, FRAUD…), source_system.
Clés de corrélationObligatoires dès la production : identifiant de transaction / Correlation ID (lien autorisation ↔ capture), référence de remise, référence de fichier (références d'estampillage). Un événement sans ses clés de corrélation est rejeté à la frontière — voir §1.3.3, c'est la clef de voûte du pipeline.
TemporalitéDistinction contractuelle entre business_time (l'heure métier de l'événement) et produced_at (l'heure de publication). Toute la logique temporelle du socle s'ancre sur business_time.
QualitéComplétude des champs obligatoires, unicité de event_id, fraîcheur (SLA de latence de publication après l'événement métier), ordre de publication par clé.
Canal d'erreurComportement en cas de rejet : DLQ dédiée par producteur, rapport de non-conformité, boucle de remédiation (cf. §1.3.4).

La distinction fondamentale : contrat d'ingestion ≠ modèle organisé

Le contrat d'ingestion capture la donnée dans la sémantique du producteur : le vocabulaire E-Stream (inspiré ISO 20022), les codes K2, les champs NEPTING. Le modèle organisé exprime la donnée dans le langage métier de l'acquisition, unique et indépendant des producteurs (cf. §2.1). Entre les deux se trouve une couche de traduction explicite — le mapping d'ingestion — elle aussi versionnée et testée.

Sémantique PRODUCTEUR TRADUCTION Sémantique MÉTIER ┌────────────────────────┐ ┌─────────────────────┐ ┌─────────────────────────┐ │ CONTRAT D'INGESTION │ │ MAPPING D'INGESTION │ │ MODÈLE ORGANISÉ │ │ (Avro, par producteur)│──▶│ (anti-corruption │──▶│ (pivot acquisition, │ │ E-Stream / K2 / NEPT. │ │ layer, versionné, │ │ unique, versionné) │ │ │ │ golden tests) │ │ │ └────────────────────────┘ └─────────────────────┘ └─────────────────────────┘
Anti-pattern à proscrire — la fuite sémantique

Si les champs et concepts des producteurs fuient directement dans le modèle organisé (« le champ 422 d'E-Stream », « le code retour K2 »), le modèle devient l'union des formats des producteurs : chaque nouveau producteur le déforme, chaque évolution amont le casse, et les consommateurs doivent connaître les systèmes sources pour l'interpréter. Le contrat absorbe la sémantique producteur ; le mapping la traduit ; le modèle organisé l'ignore. La provenance est conservée uniquement dans des champs techniques de lineage (source_system, source_message_ref, ingestion_schema_version), jamais dans la sémantique métier.

Gouvernance du contrat

  • Le contrat est du code : fichiers .avsc dans Git, revue obligatoire, CI qui valide la compatibilité contre le Registry avant tout déploiement producteur.
  • Golden samples : chaque producteur fournit un jeu de messages de référence (cas nominaux + cas limites). Ces échantillons alimentent les tests automatiques du mapping d'ingestion.
  • Évolution : tout changement de schéma suit la procédure — proposition, revue plateforme + producteur, publication d'une nouvelle version compatible, dépréciation planifiée des champs retirés.
  • Le Business Analyst facilite, la CI valide : le BA anime la rédaction du contrat avec le producteur et l'ingénierie ; la conformité n'est plus une affaire de prose Confluence mais de tests exécutés à chaque build.

1.2 Pipeline 1 — Stockage Raw : Kafka (Avro/JSON) → Parquet / Iceberg

Objectif : capturer tous les événements, tels que reçus, dans un stockage immuable, peu coûteux, requêtable en SQL et rejouable. Le Raw Lake est le filet de sécurité de la plateforme : toute projection (dont le modèle organisé lui-même) peut être reconstruite depuis lui. Format de fichiers : Parquet ; format de table : Apache Iceberg.

1.2.1 Des messages unitaires aux fichiers Parquet — le problème et sa résolution

Parquet est un format colonnaire par blocs : il est conçu pour être écrit par gros lots et lu par colonnes, pas pour être écrit message par message. Écrire un fichier par événement produirait des millions de « small files » qui détruisent les performances de lecture (chaque fichier = un open/close S3, des métadonnées, un footer) et font exploser les métadonnées Iceberg. Le passage du flux unitaire Kafka au fichier Parquet repose donc sur un tampon d'accumulation :

Kafka (messages unitaires Avro) │ poll() en continu ▼ ┌──────────────────────────────────────────────────────┐ │ WRITER (Kafka Connect / Flink) │ │ │ │ 1. Désérialisation Avro (Schema Registry) │ │ 2. Routage vers la table/partition Iceberg cible │ │ (par event_type, par jour de business_time) │ │ 3. ACCUMULATION en mémoire / fichiers ouverts │ │ un writer Parquet ouvert par (table, partition) │ │ 4. FLUSH quand seuil atteint : │ │ - taille cible du fichier (128–512 Mo) │ │ - OU fin d'intervalle de commit (2–5 min) │ │ → data files Parquet fermés, prêts à être commités │ └──────────────────────────────────────────────────────┘ │ ▼ COMMIT ICEBERG (atomique, cf. §1.2.2) Table Iceberg — nouveaux fichiers visibles d'un coup

Deux implémentations de référence :

OptionFonctionnementQuand la choisir
Kafka Connect —
Iceberg Sink Connector		 Les workers Connect consomment, bufferisent et écrivent les data files Parquet. Un coordinateur de commit agrège les fichiers produits par tous les workers et réalise un commit Iceberg unique à intervalle fixe (iceberg.control.commit.interval-ms, typ. 120 000–300 000 ms). Les offsets Kafka sont consignés dans les propriétés du snapshot : en cas de crash, le connector reprend depuis le dernier commit → sémantique exactly-once. Supporte le fan-out multi-tables (router chaque event_type vers sa table) et la création automatique de tables avec évolution de schéma. Recommandé pour le raw : aucune transformation, configuration déclarative, pas de code à maintenir.
Flink — Iceberg Sink Les fichiers sont écrits en continu ; le commit Iceberg est aligné sur les checkpoints Flink (exactly-once bout en bout via two-phase commit). Latence et fréquence de commit pilotées par l'intervalle de checkpoint. Si un enrichissement ou une normalisation en vol devient nécessaire avant le raw (à éviter par principe : le raw doit rester brut).
Règle — la fréquence de commit est un compromis fraîcheur / small-files

Commiter toutes les 2 à 5 minutes donne un lake frais à quelques minutes — largement suffisant pour un raw analytique — tout en produisant des fichiers de taille saine. Ne jamais chercher la « fraîcheur seconde » sur le lake : c'est le rôle du pipeline MongoDB (§1.3). Un commit par message ou par petit lot est l'erreur classique qui rend une table Iceberg inexploitable en six mois.

Enveloppe raw — schéma des tables

Chaque table raw porte une enveloppe standard, identique pour tous les types d'événements, plus le payload dans la sémantique du producteur (le raw ne traduit rien) :

event_id string — identité unique (idempotence, dédoublonnage)
event_type string — AUTHORIZATION | CAPTURE | REMITTANCE | CHARGEBACK | FRAUD | …
source_system string — ESTREAM | K2 | NEPTING | COMPUTOP | BATCH_COS
schema_version string — version du contrat d'ingestion appliqué
business_time timestamp — heure métier (colonne de partitionnement)
produced_at timestamp — heure de publication Kafka
ingested_at timestamp — heure d'écriture lake
correlation struct — { transaction_id, correlation_id, remittance_ref, file_ref, siren, rib, store_id }
kafka struct — { topic, partition, offset } — traçabilité de rejeu
payload struct — le message producteur, intégral, non traduit

1.2.2 Mécanique de commit Iceberg et mise à jour du catalogue

Comprendre la mécanique interne d'Iceberg est indispensable pour dimensionner la maintenance. Une table Iceberg est un arbre de métadonnées immuables pointé par le catalogue :

CATALOGUE (REST / Nessie / Glue / Hive Metastore) │ ne stocke qu'UNE chose par table : │ le pointeur vers le metadata.json courant ▼ metadata.json (vN) ── schéma, partition spec, historique des snapshots │ ▼ manifest-list (du snapshot courant) ── liste des manifests + stats de partition │ ├──▶ manifest-file A ── liste de data files + stats min/max par colonne │ ├──▶ data-file-001.parquet │ └──▶ data-file-002.parquet └──▶ manifest-file B └──▶ data-file-003.parquet UN COMMIT = 1. écrire les nouveaux data files Parquet (invisible pour les lecteurs) 2. écrire un nouveau manifest les référençant (invisible) 3. écrire une nouvelle manifest-list = snapshot N+1 (invisible) 4. écrire metadata.json vN+1 (invisible) 5. CAS atomique sur le catalogue : « remplace le pointeur vN par vN+1 si le courant est toujours vN » → concurrence optimiste : en cas de conflit, relecture + retry → les lecteurs voient l'ancien snapshot OU le nouveau, jamais un état partiel

Conséquences pratiques :

  • Snapshot isolation gratuite : une requête Starburst démarrée pendant un commit lit un snapshot cohérent de bout en bout.
  • Time travel natif : chaque commit est un snapshot horodaté ; on peut requêter la table « telle qu'elle était » à un instant passé — précieux pour l'audit et la réconciliation comptable.
  • Le catalogue est critique mais léger : il ne fait que le CAS du pointeur. Recommandation : un catalogue REST (standard d'interopérabilité 2025+, supporté par Starburst, Spark, Flink, Kafka Connect) adossé à un backend HA.
  • Exactly-once du connector : les offsets Kafka étant stockés dans les propriétés du snapshot, offset-commit et data-commit sont le même acte atomique — pas de fenêtre de double écriture ni de perte.

1.2.3 Stratégie de partitionnement Iceberg

Iceberg pratique le hidden partitioning : la partition est déclarée comme une transformation d'une colonne (days(business_time)), les fichiers sont physiquement organisés selon cette transformation, et les requêtes filtrant sur business_time bénéficient du pruning sans jamais manipuler de colonne de partition. C'est un avantage décisif sur le partitionnement Hive (pas de colonne technique dt=2026-07-01 à connaître, pas d'erreur de requête possible).

1.2.4 Maintenance des tables — non optionnelle

Un flux continu produit mécaniquement des fichiers plus petits que l'optimal et une accumulation de snapshots. La maintenance est un composant de l'architecture, planifié dès le premier jour :

ProcédureRôleCadence recommandée
rewrite_data_filesCompaction : fusionne les petits fichiers vers la taille cible (256–512 Mo), applique le sort-order.Quotidienne, sur la partition J-1 (les partitions closes ne bougent plus).
expire_snapshotsPurge les snapshots anciens et les fichiers qu'ils étaient seuls à référencer. Borne le time travel.Quotidienne — rétention 7 jours de snapshots (le rejeu long terme se fait sur l'état courant, pas sur les snapshots).

remove_orphan_filesSupprime les fichiers écrits mais jamais commités (crashs de writers).Hebdomadaire, avec garde de sécurité > 3 jours.
Ce que le Raw Lake apporte au reste de l'étude

Rejouabilité totale : le pipeline MongoDB peut être reconstruit, un bug de mapping peut être corrigé puis rejoué, et surtout — c'est l'argument central du §2.4 — la v2 du modèle organisé peut être alimentée par re-projection depuis le raw, plutôt que par une migration base-à-base qui transporterait les défauts de la v1.

1.3 Pipeline 2 — Alimentation de la base transactionnelle MongoDB (modèle organisé)

Ce pipeline transforme le flux d'événements en documents du modèle organisé : upserts corrélés dans les collections MongoDB, à fort débit, avec une exigence de fraîcheur bien supérieure au lake (secondes à minutes). Sa performance repose sur trois décisions prises en amont du consumer : la clé de production, le nombre de partitions, et la corrélation portée par l'événement.

1.3.1 Choix des clés de production Kafka — une décision d'architecture, pas un détail

La clé du message Kafka détermine la partition d'affectation, donc l'ordre de traitement et la localité : tous les messages d'une même clé arrivent ordonnés sur la même partition et sont traités par le même consumer. C'est cette propriété qui permet des écritures MongoDB sans lecture préalable (§1.3.3).

Candidat de cléVerdictAnalyse
transaction_id /
correlation_id		RECOMMANDÉTous les mouvements d'une même transaction (autorisation, capture, compensation, chargeback) convergent ordonnés vers la même partition → le consumer voit le cycle de vie d'une transaction séquentiellement, les upserts sur le document cible ne se concurrencent jamais entre partitions. Cardinalité très élevée (dizaines de millions) → répartition statistiquement uniforme.
store_id / merchant_idÀ PROSCRIRECrée des hot partitions : un grand commerçant (type Castorama, dizaines de milliers de transactions/jour) concentre son trafic sur une seule partition, qui devient le goulot du pipeline entier pendant que les autres sont vides.

Exigence contractuelle envers les producteurs

La clé de production fait partie du contrat d'ingestion (§1.1). Un producteur qui publie avec une mauvaise clé — ou sans clé — casse silencieusement l'ordre et la localité : le défaut n'apparaît qu'en charge, sous forme de documents incohérents. La clé exigée, sa provenance dans le payload et sa cardinalité sont donc validées à la revue de contrat, et monitorées (distribution des messages par partition).

1.3.2 Dimensionnement du nombre de partitions — large, dès la création

Le nombre de partitions borne le parallélisme maximal des consumers (au-delà, les consumers sont inactifs) et se décide à la création du topic. Règles de dimensionnement :

Débit de dimensionnement : pic soutenu × facteur de reprise (backlog replay) ≈ 3–5×
Capacité prudente/partition : 5–10 Mo/s ou quelques milliers de msg/s (selon taille des messages)
Parallélisme consumer : nb consumers actifs max = nb partitions

Exemple CPDP — topic transactions enrichies :
pic estimé 2 000 msg/s × reprise 4× = 8 000 msg/s cible
÷ ~500 msg/s de traitement MongoDB par consumer thread (upserts batchés)
→ 16 consumers minimum → 64 partitions retenues (marge de croissance ×4)

Recommandation : 64 partitions minimum sur les topics transactionnels,
128 sur les topics à plus forte volumétrie (autorisations).
Pourquoi surdimensionner d'emblée — l'augmentation ultérieure casse l'affectation par clé

Augmenter le nombre de partitions d'un topic existant change le résultat de hash(clé) % nb_partitions : les événements d'une même transaction, jusque-là routés vers la partition 12, partent désormais vers la partition 47 — pendant que d'anciens événements de la même clé restent en attente sur la partition 12. L'ordre par clé est rompu de façon transitoire, précisément la garantie sur laquelle repose tout le pipeline. La seule stratégie saine : dimensionner large à la création (le coût d'une partition surnuméraire est négligeable), et considérer un repartitionnement ultérieur comme une migration à part entière (nouveau topic + bascule contrôlée).

1.3.3 La corrélation sécurisée à l'ingestion — jamais au processing

C'est la décision la plus structurante du pipeline. Deux philosophies s'opposent :

── ANTI-PATTERN : corrélation AU PROCESSING (lookup bloquant) ────────────────── pour chaque événement consommé : parent = mongodb.findOne({ ... }) ← appel réseau UNITAIRE, BLOQUANT si parent trouvé : rattacher, écrire sinon : réessayer plus tard / mettre de côté Conséquences mesurables : · ~1–3 ms de RTT par lookup × millions d'événements/jour · débit consumer plafonné par la latence MongoDB, pas par Kafka · le lag s'accumule dès le premier ralentissement de la base (backpressure) · à la reprise d'un backlog : effondrement (chaque événement rejoué re-paye le lookup) ── PATTERN CIBLE : corrélation À L'INGESTION (portée par l'événement) ────────── le contrat (§1.1) garantit que l'événement PORTE ses clés : transaction_id, correlation_id, remittance_ref, file_ref le consumer écrit alors en AVEUGLE, sans aucune lecture préalable : bulkWrite(ordered=false) de updateOne( filter : { _id: transaction_id }, update : { $setOnInsert: {squelette}, $set: {champs}, $push / $set positionnel : mouvement }, upsert : true ) · zéro lecture dans le chemin chaud — le débit est celui de bulkWrite batché · idempotent et rejouable (event_id + gardes de version type batch_session) · l'ordre par clé (§1.3.1) élimine les courses entre consumers

Le cas des événements hors ordre (une capture qui arrive avant son autorisation, un chargeback avant que la transaction n'existe) est absorbé par le même mécanisme : l'upsert $setOnInsert crée un document squelette portant l'identité et le mouvement reçu ; l'arrivée ultérieure du parent complète le squelette. À aucun moment le consumer n'attend, ne relit, ni ne bloque.

Traitement contractuel de la corrélation avec les producteurs

La corrélation devient ainsi une obligation du producteur, pas un service de la plateforme : E-Stream fournit le Correlation ID reliant autorisation et capture ; les références d'estampillage relient transaction, remise et fichier ; K2 et NEPTING fournissent leurs identités de transaction. La plateforme vérifie (rejet en DLQ si clés absentes) et assemble (upserts), elle ne devine jamais. Toute règle de corrélation « dynamique » qui exigerait de retrouver le parent par heuristique est un défaut de contrat à remonter au producteur — pas un algorithme à implémenter dans le consumer.

1.3.4 Consolidation désynchronisée et détection proactive des orphelins

Malgré les contrats, le réel produira des orphelins : mouvements dont le parent n'arrive jamais (perte amont), clés présentes mais incohérentes, cas limites de reprise. Le principe : ces cas sont traités hors du chemin chaud, par des processus asynchrones, et rendus visibles plutôt que silencieusement accumulés.

Chemin chaud (consumers Kafka) Chemins asynchrones ┌────────────────────────────────┐ │ upserts aveugles idempotents │ │ → documents complets │ │ → documents SQUELETTES │──┐ │ (parent pas encore arrivé) │ │ │ → DLQ (contrat violé : │ │ ┌─────────────────────────────────┐ │ clés manquantes, schéma KO) │──┼──▶│ CONSOLIDATEUR (job périodique) │ └────────────────────────────────┘ │ │ · scan batché des squelettes │ │ │ (index sur {status:"stub", │ │ │ first_seen}) — jamais unitaire│ └──▶│ · tentatives de rattachement │ │ · re-émission des DLQ corrigées │ └───────────────┬─────────────────┘ │ au-delà des seuils ▼ ┌─────────────────────────────────┐ │ DÉTECTION PROACTIVE │ │ · métriques : nb orphelins, │ │ âge du plus ancien, taux de │ │ corrélation — PAR PRODUCTEUR │ │ · seuils : ex. orphelin > 24 h, │ │ taux < 99,9 % sur 1 h │ │ · RAPPORT DE NON-CORRÉLATION │ │ notifié au producteur │ │ (event_ids, clés, périodes) │ │ · suivi de remédiation │ └─────────────────────────────────┘
  • Le consolidateur travaille par balayages batchés indexés (jamais de lookup unitaire), à cadence régulière (ex. toutes les 10 minutes), et n'entre jamais en concurrence avec le chemin chaud sur les mêmes documents grâce aux upserts idempotents.
  • Les métriques d'orphelins sont des indicateurs de qualité producteur, publiés par producteur et par type d'événement : elles objectivent le respect du contrat et déclenchent la boucle de remédiation (notification outillée, pas mail artisanal).
  • Rien ne disparaît : un orphelin définitif reste requêtable (raw + collection de quarantaine), horodaté, imputable — exigence d'audit sur des données financières.

2. Le modèle organisé

2.1 Indépendance sémantique vis-à-vis des producteurs

Le modèle organisé est le langage métier unique de l'acquisition. Il décrit des concepts du domaine — la transaction et son cycle de vie, les mouvements (autorisation, capture, compensation), la remise, le chargeback, la fraude — et non des formats de flux. E-Stream, K2, NEPTING et demain Computop sont des dialectes d'entrée que la couche de mapping (§1.1) traduit vers ce pivot.

Ce principe a trois conséquences concrètes :

  • Ajouter un producteur = écrire un mapping, jamais modifier le modèle. Le raccordement de Computop est un projet d'intégration (contrat + mapping + tests), pas un projet de re-modélisation.
  • Test de convergence multi-sources : une même transaction observée par K2 et par E-Stream doit produire, après mapping, le même document organisé — aux champs de lineage près. Ce test (golden tests croisés) est le critère objectif d'indépendance sémantique, exécuté en CI.
  • Lineage sans pollution : la provenance (source_system, source_message_ref, ingestion_schema_version, offsets Kafka) vit dans une section technique _lineage de chaque document. Elle sert l'audit et le débogage ; aucun consommateur métier n'a besoin de la lire pour interpréter la donnée.
Position d'architecte — sur la dérivation depuis le modèle de transfert E-Stream

Le modèle actuel a été largement déduit du modèle de transfert E-Stream (lui-même partiellement inspiré d'ISO 20022). Un modèle de transport est optimisé pour échanger des messages, pas pour représenter un domaine ni servir des lectures : en le prenant comme point de départ, on hérite de contraintes de messagerie dans le cœur de données. Le choix de ne pas adopter intégralement un standard (ISO 20022 complet, BIAN) pour des raisons de coût est entendable et assumé ; en revanche, s'inspirer de leur structure conceptuelle (le vocabulaire, la séparation transaction / mouvements / règlement) ne coûte rien et évite de réinventer un domaine que l'industrie a déjà éprouvé. C'est exactement la cible v2 : un modèle conçu pour le métier et les patterns de lecture, mappé depuis E-Stream — et non déduit de lui.

2.3 Modèle cible v2 — le métier de l'acquisition

La v2 décrit proprement le domaine, en tirant les leçons de la v1 :

┌───────────────────────────────────────────────┐ ┌──────────────────────────┐ │ TRANSACTION (agrégat, collection) │ │ REMISE (collection) │ │ _id = transaction_id (clé de corrélation) │ │ _id = remittance_ref │ │ identité : store_id, merchant, siren, rib │ │ objet INDÉPENDANT : │ │ montants consolidés, devise │ │ ses propres montants, │ │ business_dates clés du cycle de vie │ │ sessions batch (1..9), │ │ │ │ lien fichier (file_ref) │ │ movements[] — MOUVEMENTS TYPÉS, ordonnés : │◀────▶│ │ │ { type: AUTHORIZATION|CAPTURE|CLEARING| │ lien │ postulat 1↔1 par tx, │ │ CHARGEBACK|FRAUD_ALERT, │ par │ cardinalité à réexaminer │ │ movement_id, correlation_id, │ clés │ (variantes pressenties) │ │ business_time, amounts, status, │ └──────────────────────────┘ │ remittance_ref?, scheme, _lineage } │ │ │ Chargeback & fraude = │ _lineage : source_system(s), refs, offsets │ mouvements du cycle de vie, └───────────────────────────────────────────────┘ rattachés par correlation_id
  • La transaction est un agrégat de mouvements typés, pas un état à statuts : l'historique est natif, les corrections rétroactives (chargeback tardif) sont des mouvements ajoutés, jamais des écrasements.
  • La remise est un objet indépendant, lié par les références d'estampillage — jamais imbriquée. Ses caractéristiques propres (sessions batch, montants bruts débit/crédit, lien fichier) vivent chez elle ; l'enrichissement éventuel de champs remise au niveau transaction est une décision du modèle logique, pas du modèle conceptuel.
  • Les clés de corrélation sont de premier rang : indexées, obligatoires, documentées champ par champ (quelle clé relie quoi, avec quelle cardinalité).
  • Le versioning du modèle (et non des flux) est outillé : schema_version dans chaque document, changelog publié, visualisation du modèle et dictionnaire des champs (signification, valeurs admises) maintenus avec le schéma — pas dans des pages orphelines.

2.4 Stratégie de migration v1 → v2 sans rupture de service

Deux stratégies sont viables ; le choix dépend de la nature du delta entre v1 et v2.

Option A — Évolution incrémentale in place (expand → migrate → contract)

Faire muter progressivement le schéma v1 jusqu'à devenir la v2, dans les mêmes collections. MongoDB s'y prête (documents versionnés par schema_version, lecteurs tolérants aux deux formes) :

  1. Expand — ajouter les structures cibles (mouvements typés, références de remise) à côté de l'existant ; les écritures alimentent les deux formes.
  2. Migrate — backfill des documents historiques vers la forme cible, par lots, en tâche de fond.
  3. Contract — retirer les champs et comportements v1 une fois tous les lecteurs migrés.

Forces : pas de double base, pas de bascule d'API, coût d'infrastructure nul. Limites : chaque étape intermédiaire doit rester lisible par l'API v1 → le code de lecture gère N formes simultanées ; quand le delta est structurel (externaliser la remise, retyper les mouvements, abandonner les statuts), la trajectoire devient longue, fragile, et le risque d'enlisement à mi-chemin est réel — l'état intermédiaire permanent est le pire des états.

Option B — v2 en parallèle, double alimentation, API v2, migration progressive des consommateurs

KAFKA (topics d'ingestion) RAW LAKE ICEBERG │ │ │ consumer group consumer group │ backfill historique organized-v1 organized-v2 ◀───── re-projection ───┘ (batch Spark/Flink, │ │ depuis le raw mêmes mappings) ▼ ▼ ┌──────────────┐ ┌──────────────┐ │ BASE v1 │ │ BASE v2 │ double alimentation au fil de l'eau │ (inchangée) │ │ (cible) │ (deux consumer groups indépendants) └──────┬───────┘ └──────┬───────┘ │ │ API v1 API v2 contrat de non-rupture : v1 figée, │ │ v2 documentée, matrice de compat publiée consommateurs ──────▶ migration progressive, consommateur par consommateur │ │ ▼ ▼ RÉCONCILIATION CONTINUE v1 ↔ v2 : comptages par jour/type/source, checksums d'agrégats (montants), échantillonnage de documents │ ▼ critères de bascule atteints → gel v1 → décommissionnement
  1. Créer la base v2 avec le modèle cible propre (§2.3) — sans compromis hérités.
  2. Alimenter la v2 par re-projection depuis le raw pour l'historique (backfill Iceberg → mappings v2 → MongoDB) et par un consumer group dédié pour le fil de l'eau. Point non négociable : pas de migration ETL base v1 → base v2, qui transporterait dans la v2 les défauts et pertes de la v1. La re-projection valide au passage la promesse de rejouabilité du socle — si elle échoue, c'est un signal architectural majeur à traiter.
  3. Double alimentation v1 + v2 pendant toute la coexistence : deux consumer groups indépendants sur les mêmes topics, aucune dépendance entre eux.
  4. Exposer l'API v2 ; l'API v1 est figée (corrections de sécurité uniquement) — le gel est ce qui rend la fin de vie crédible.
  5. Migrer les consommateurs progressivement, avec la réconciliation continue v1 ↔ v2 comme preuve de confiance (les chiffres sont identiques, documentés, opposables).
  6. Décommissionner sur critères explicites : 100 % du trafic sur v2 pendant N semaines, réconciliation verte, plan de retour arrière expiré.
Recommandation d'architecte

Le delta v1 → v2 est structurel (mouvements typés, remise externalisée, abandon des statuts, convergence tri-source K2/NEPTING/E-Stream) : l'Option B est recommandée, avec alimentation de la v2 par re-projection depuis le raw. L'Option A reste l'outil de droit commun pour les évolutions additives ultérieures de la v2 (nouveaux champs, nouveaux types de mouvements) — c'est la combinaison des deux qui constitue la stratégie complète : B pour changer de génération, A pour vivre à l'intérieur d'une génération. Condition de réussite commune : le pipeline raw du §1.2 doit être en production avant d'engager la migration, puisqu'il en est le moteur.
