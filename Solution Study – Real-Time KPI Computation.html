1Business Context and Requirements

1.1 Operational Setting

This document studies the architecture of a real-time KPI computation system for a large-scale payment acquiring platform. The platform aggregates transaction flows from thousands of merchants, each operating one or more points of sale (POS). Transactions originate from payment processors, card scheme acquirers, and acceptance terminals, and are ingested as high-throughput event streams via Apache Kafka.

The existing processing pipeline consumes raw transaction events from payment.transaction.raw.v1, performs reconciliation and correlation with reference data stored in MongoDB, and publishes fully enriched events onto payment.transaction.enriched.v1. The new KPI layer attaches to this enriched stream as an independent consumer group, without modifying any existing code.

1.2 Business Need — KPI Catalogue

Merchants require near real-time visibility into their payment activity (activité monétique) through a dedicated monitoring dashboard. The following KPIs must be computed continuously, partitioned by point of sale, by currency, and by business day:

KPITypeDefinition
Total collected (gross)SumSum of all approved transaction amounts for the business day
Total refundsSumSum of all refund amounts
Net collectedDerivedgross_amount − refund_amount
Transaction countCountApproved transactions (success_count), failed transactions (failed_count), total
Average basketDerivedgross_amount / success_count — computed at read time from stored sufficients
Failure rateDerivedfailed_count / total_count × 100
Fraud indicatorsCount + SumFraud count, fraud amount (optional scope)

Design principle — statistical sufficients

The accumulator stores only raw sums and counts (gross_amount, refund_amount, success_count, failed_count, total_count). Derived metrics (average basket, failure rate, net amount) are computed at read time. This ensures that any late-arriving event that increments the stored sufficients automatically produces a correct derived value without any special correction logic.

1.3 Architectural Starting Point

The platform already operates a Kafka Streams topology consuming payment.transaction.raw.v1. This pipeline performs event correlation, deduplication, and enrichment, then publishes canonical events onto payment.transaction.enriched.v1, partitioned by merchant_id. The new KPI computation layer must attach to this topic as an independent consumer group (application.id: kpi-processor) and must not introduce any dependency on, or modification to, the existing pipeline.

2Functional and Non-Functional Constraints

2.1 Functional Constraints

2.1.1 KPI Semantics — Simple Aggregates per POS, per Currency, per Business Day

KPIs are simple aggregates (sums, counts, derived ratios) scoped to a single (store_id, business_date, currency) triplet. This bounded scope has important architectural consequences: the state per key is compact and can be maintained in a local embedded store without cross-partition coordination. No complex event patterns, streaming joins, or ML inference are required at computation time.

2.1.2 Business Timestamp vs. Ingestion Timestamp

Every enriched transaction carries a field business_timestamp, representing the moment the transaction was initiated at the terminal. This field is produced by the payment processor and embedded in the event body. All KPI aggregation — including the derivation of business_date for the composite key — must rely exclusively on business_timestamp, not on the Kafka record timestamp (which reflects ingestion time).

Critical constraint — event time vs. processing time

The lag between business_timestamp and Kafka ingestion time can range from seconds to several days in production, due to terminal offline operation, acquirer batch settlement windows, or network partitions. A transaction with business_timestamp of two days ago received today must be attributed to the KPI partition for that past business day, not to today.

2.1.3 Late Event Integration and Retroactive Correction

The system must accept late-arriving events — events whose business_timestamp falls on a past business day — and integrate them correctly into the corresponding historical KPI partition. There is no immutable window closure: the KPI state for any business day remains open and updatable for a configurable retention period (default: 90 days). When a past KPI accumulator is updated, a corrected snapshot is published on both output topics with an explicit is_retroactive_correction: true flag.

Why native Kafka Streams windowing cannot be used

Kafka Streams TimeWindows and WindowStore close windows after a configurable grace period (typically minutes), after which late events are silently discarded. In a payment acquiring context, late arrivals from acquirer batch settlements or offline terminals routinely arrive hours or days late. Using native windowing would silently corrupt historical KPIs with no recovery path. The solution replaces windowing with an open KeyValueStore keyed by composite key, described in section 4.1.2.

2.1.4 Cross-POS Consolidation at Query Time

Merchants may own multiple POS terminals. The system must support on-demand aggregation of KPIs across all POS terminals belonging to a given merchant. This consolidation is performed at query time by the REST API, not pre-computed in the streaming layer. The API retrieves individual POS accumulators from the read model database and merges them, recomputing derived ratios from the merged sums and counts. No streaming-layer topology change is required to add this capability.

2.1.5 Idempotency and Deduplication

The platform's Kafka producers operate with at-least-once delivery. The system must handle duplicate events and replay scenarios. Two complementary layers of protection are applied:

  • Kafka EOS (exactly_once_v2): atomic offset commits and transactional producers prevent duplicate processing at the Kafka Streams layer for infrastructure-level redeliveries.
  • Application-level deduplication: the accumulator maintains a Set<String> of already-processed event_id values. Any event whose event_id is already present in the set is silently dropped, protecting against upstream retry logic at the acquirer or processor level that is invisible to Kafka's EOS mechanism.

2.2 Non-Functional Constraints

2.2.1 Data Freshness — Dual SLA

The system operates two distinct freshness channels with different latency targets:

ChannelTopicTarget latencyConsumer
Real-time pushkpi.store.day.current.v1< 1–2 secondsPush service → WebSocket / SSE → dashboard
Persistent storekpi.store.day.snapshot.v11–10 minutesPersistence consumer → read model DB → REST API

This separation avoids the write amplification that would result from persisting every intermediate accumulator update to the database. The real-time channel uses the full-fidelity KTable output; the persistent channel uses a throttled, debounced stream described in section 4.1.8.

2.2.2 Decoupling from the Primary Processing Pipeline

  • Independent consumer group: kpi-processor maintains its own committed offsets on payment.transaction.enriched.v1, with no dependency on the reconciliation pipeline's offsets.
  • No shared state: the two topologies share no state store, changelog topic, or application instance.
  • Failure isolation: a failure, rebalance, or restart in the KPI topology has no impact on the reconciliation pipeline, and vice versa.
  • No upstream modifications: the existing pipeline code is not changed. The KPI layer is additive only.

2.2.3 Exactly-Once Semantics

The KPI Streams Processor is configured with processing.guarantee=exactly_once_v2. This ensures that each event impacts KPI aggregates exactly once at the infrastructure level. Combined with application-level event_id deduplication, the system provides the strongest available correctness guarantee: no double-counting, no data loss, even under failure and replay conditions.

2.2.4 Fault Tolerance and State Recovery

The KPI KeyValueStore (RocksDB) is continuously backed up to a Kafka changelog topic (kpi-processor-kpi-store-pos-date-changelog). On instance failure or restart, the recovering instance replays its changelog to rebuild local state without reprocessing the full transaction history. Recovery time is proportional to state changes since the last checkpoint, not to total transaction volume. The changelog topic must be retained for at least the same duration as the KPI store retention period (90 days).

2.2.5 Write Load Control on the Exposure Database

Without throttling, each transaction event would trigger an accumulator update that would propagate as a database write — unsustainable at scale. The system applies a Dirty Key Tracking / Debounced Sink pattern: the KTable output is passed through a Suppressed.untilTimeLimit() operator that absorbs intermediate updates per key and emits only the latest accumulated value after a configurable silence window (default: 5 minutes). This reduces database write load by a factor equal to the average number of transactions per POS per throttle window.

2.2.6 Scalability

Scalability is achieved by increasing the number of partitions on payment.transaction.enriched.v1 and deploying additional Kafka Streams instances. Each instance processes a disjoint set of partitions with a fully independent local state store — no cross-instance coordination is required for KPI computation. The partition key must be store_id (or merchant_id) to guarantee that all events for a given point of sale land on the same partition and are processed by the same instance, enabling correct local aggregation.

3Solution Comparison: Streaming vs. Materialized Views

3.1 Candidate Approaches

Option A — Stream Processing (Kafka Streams) SELECTED

KPIs are computed incrementally as each enriched transaction event is consumed. An embedded RocksDB state store maintains a running accumulator per (store_id, business_date, currency) composite key. Two output topics are produced: a full-fidelity topic for real-time push, and a debounced topic for database persistence. The read model database is updated via a batch persistence consumer, decoupled from the streaming computation.

Option B — Materialized Views over MongoDB

Enriched transactions are written to MongoDB by the existing reconciliation pipeline. KPIs are computed by executing $group / $sum / $avg aggregation pipelines over the transactions collection. Results are stored in a separate KPI collection refreshed periodically (scheduled job or Change Stream trigger).

3.3 Decision

Selected approach: Option A — Apache Kafka Streams
  1. Data freshness: sub-second updates on the real-time channel satisfy the push SLA without compromise; the debounced channel satisfies the DB persistence SLA (1–10 min).
  2. Late event correctness: the open KeyValueStore keyed by (store_id|business_date|currency) handles retroactive corrections natively for up to 90 days — impossible with native windowing or materialized views.
  3. Write load control: the Dirty Key Tracking / Debounced Sink pattern (Suppress operator) reduces DB write amplification by an order of magnitude at peak load.
  4. Decoupling: the KPI topology is a pure reader of the enriched topic; it shares no resources with the reconciliation pipeline.
  5. Exactly-once: EOS v2 combined with application-level event_id deduplication provides the strongest available correctness guarantee.
  6. Infrastructure reuse: the solution builds entirely on the existing Kafka infrastructure; no new compute technology is required.

4Solution Architecture — Apache Kafka Streams

The solution is organized into two subsystems: the KPI computation and storage layer (Kafka Streams topology, RocksDB state stores, output topics) and the KPI exposure layer (persistence consumer, read model database, REST API, WebSocket push).

4.1 KPI Computation and Storage

4.1.1 Input Stream Contract

The topology consumes payment.transaction.enriched.v1, partitioned by store_id. The following fields are required:

JSON — payment.transaction.enriched.v1 payload
{
  "event_id":            "evt-abc123",        // globally unique — used for deduplication
  "store_id":            "pos-789",           // point of sale identifier (= partition key)
  "merchant_id":         "merchant-456",
  "business_timestamp":  "2024-03-15T09:32:11.000Z", // BUSINESS time from payload
  "amount_cents":        4250,               // integer — avoids float precision issues
  "currency":            "EUR",
  "status":              "APPROVED",         // APPROVED | DECLINED | ERROR | REFUND
  "event_type":          "PAYMENT",          // PAYMENT | REFUND
  "is_fraud":            false
}

4.1.2 Composite Key Design — Open Partitions Instead of Windows

Rather than using Kafka Streams native windowing (whose window closure discards late events), the topology re-keys each event using a composite key that encodes the point of sale, the business date extracted from business_timestamp, and the currency. The KeyValueStore then acts as an open dictionary — any event with any past business date updates the correct accumulator without any window closing.

Java — composite key derivation
KStream<String, EnrichedTransaction> rekeyed = valid
    .selectKey((oldKey, tx) -> {
        // Business date from the PAYLOAD field — never from system clock or Kafka timestamp
        LocalDate businessDate = tx.getBusinessTimestamp()
            .atZone(ZoneOffset.UTC)
            .toLocalDate();

        // Composite key: store_id|YYYY-MM-DD|currency
        // Example: "pos-789|2024-03-14|EUR"
        return tx.getStoreId() + "|" + businessDate + "|" + tx.getCurrency();
    });
Why currency is part of the key

An acquiring platform processing multi-currency merchants must avoid mixing amounts from different currencies in the same accumulator. Including currency in the composite key ensures that each sum and count is always currency-homogeneous. Cross-POS consolidation at query time only merges accumulators with the same currency, or performs conversion explicitly.

4.1.3 Business Timestamp Extractor

A custom TimestampExtractor reads business_timestamp from the event payload, ensuring that all temporal logic within the topology operates on business time:

Java — TransactionEventTimeExtractor
public class TransactionEventTimeExtractor implements TimestampExtractor {
    @Override
    public long extract(ConsumerRecord<Object, Object> record, long partitionTime) {
        if (record.value() instanceof EnrichedTransaction tx
                && tx.getBusinessTimestamp() != null) {
            return tx.getBusinessTimestamp().toEpochMilli();
        }
        return partitionTime; // fallback: Kafka record timestamp
    }
}

4.1.4 Why Native Windowing Is Not Used

Kafka Streams TimeWindows and WindowStore operate by closing windows after a configurable grace period. Any event arriving after the window is closed is silently discarded. In a payment acquiring context, this is unacceptable:

  • Acquirer batch settlements may arrive 12–48 hours after the business day
  • Offline terminals reconnecting may submit transactions from the previous day
  • Network incidents at a processor may delay a batch of events by days

The open KeyValueStore approach replaces temporal windowing with key-space partitioning: the business date is part of the composite key rather than a window boundary, so the "window" never closes as long as the key exists in the store (up to the 90-day retention limit).

4.1.5 KPI Accumulator — Statistical Sufficients and Idempotence

Java — PosKpiAccumulator
public class PosKpiAccumulator {

    // ── Statistical sufficients ───────────────────────────────────────────
    private long grossAmountCents   = 0L;   // sum of APPROVED amounts
    private long refundAmountCents  = 0L;   // sum of REFUND amounts
    private long successCount       = 0L;   // APPROVED count
    private long failedCount        = 0L;   // DECLINED + ERROR count
    private long totalCount         = 0L;   // all events
    private long fraudCount         = 0L;
    private long fraudAmountCents   = 0L;
    private Instant lastEventTimestamp;

    // ── Application-level deduplication ──────────────────────────────────
    // Protects against at-least-once redelivery AND upstream acquirer retries
    private Set<String> processedEventIds = new HashSet<>();

    public PosKpiAccumulator update(EnrichedTransaction tx) {
        if (!processedEventIds.add(tx.getEventId())) {
            return this; // idempotence gate — already processed
        }
        totalCount++;
        switch (tx.getEventType()) {
            case "PAYMENT" -> {
                switch (tx.getStatus()) {
                    case "APPROVED" -> { grossAmountCents += tx.getAmountCents(); successCount++; }
                    case "DECLINED", "ERROR" -> failedCount++;
                }
            }
            case "REFUND" -> refundAmountCents += tx.getAmountCents();
        }
        if (tx.isFraud()) { fraudCount++; fraudAmountCents += tx.getAmountCents(); }
        lastEventTimestamp = tx.getBusinessTimestamp();
        return this;
    }

    public KpiSnapshot toSnapshot(String storeId, LocalDate date, String currency) {
        // Derived metrics computed at read time — correct by construction for late arrivals
        long netAmountCents = grossAmountCents - refundAmountCents;
        double avgBasket = successCount > 0 ? (double) grossAmountCents / successCount : 0.0;
        double failureRate = totalCount > 0 ? failedCount * 100.0 / totalCount : 0.0;

        return KpiSnapshot.builder()
            .storeId(storeId).businessDate(date).currency(currency)
            .grossAmountEuros(grossAmountCents / 100.0)
            .refundAmountEuros(refundAmountCents / 100.0)
            .netAmountEuros(netAmountCents / 100.0)
            .successCount(successCount).failedCount(failedCount).totalCount(totalCount)
            .avgBasketEuros(avgBasket / 100.0)
            .failureRatePct(failureRate)
            .fraudCount(fraudCount)
            .lastEventTimestamp(lastEventTimestamp)
            .build();
    }
}

Java — Kafka Streams topology (full)
StreamsBuilder builder = new StreamsBuilder();

KStream<String, EnrichedTransaction> txStream = builder.stream(
    "payment.transaction.enriched.v1",
    Consumed.with(Serdes.String(), enrichedTxSerde)
             .withTimestampExtractor(new TransactionEventTimeExtractor()));

KStream<String, EnrichedTransaction> valid = txStream
    .filter((k, tx) -> tx.getStoreId() != null
                    && tx.getBusinessTimestamp() != null
                    && tx.getCurrency() != null);

KStream<String, EnrichedTransaction> rekeyed = valid
    .selectKey((k, tx) -> tx.getStoreId() + "|"
        + tx.getBusinessTimestamp().atZone(ZoneOffset.UTC).toLocalDate()
        + "|" + tx.getCurrency());

KTable<String, PosKpiAccumulator> kpiTable = rekeyed
    .groupByKey(Grouped.with(Serdes.String(), enrichedTxSerde))
    .aggregate(
        PosKpiAccumulator::new,
        (key, tx, acc) -> acc.update(tx),
        Materialized
            .<String, PosKpiAccumulator, KeyValueStore<Bytes, byte[]>>
            as("kpi-store-pos-date")
            .withKeySerde(Serdes.String())
            .withValueSerde(accumulatorSerde)
            .withRetention(Duration.ofDays(90)));

// ── Branch 1: real-time push (no throttle) ────────────────────────────
kpiTable.toStream()
    .map((key, acc) -> toSnapshotRecord(key, acc, false))
    .to("kpi.store.day.current.v1",
        Produced.with(Serdes.String(), snapshotSerde));

// ── Branch 2: debounced DB persistence (Dirty Key Tracking pattern) ───
kpiTable
    .suppress(
        Suppressed.untilTimeLimit(
            Duration.ofMinutes(5),
            BufferConfig.maxBytes(50 * 1024 * 1024L)  // 50 MB spill cap
                .withLoggingEnabled(Map.of(
                    "retention.ms", String.valueOf(Duration.ofHours(2).toMillis())))))
    .toStream()
    .map((key, acc) -> toSnapshotRecord(key, acc, true))  // marks as snapshot
    .to("kpi.store.day.snapshot.v1",
        Produced.with(Serdes.String(), snapshotSerde));

Log compaction ensures that for each composite key (store_id|date|currency), only the latest KPI snapshot is retained in the topic. This makes the topic usable as a lightweight system of record for bootstrapping new consumers (e.g., a new dashboard instance) without reprocessing the full transaction history.

4.1.8 Dirty Key Tracking Pattern — Throttle Mechanism Detail

The Dirty Key Tracking pattern is the conceptual name for the mechanism implemented by Suppressed.untilTimeLimit(). The name comes from the idea that a key is "dirty" (has unsaved changes) from the moment its accumulator is updated until its value is emitted downstream. Kafka Streams' Suppress operator implements this efficiently:

  • An in-memory buffer (backed by RocksDB if maxBytes is exceeded) holds the latest value per dirty key.
  • A key is emitted (flushed) only when no new update has been received for that key for the configured silence window (5 minutes by default).
  • If the buffer reaches its size cap before the silence window expires, the oldest dirty key is emitted early (spill-to-disk behaviour).
  • The result: N transaction events per POS per silence window → exactly 1 emission to kpi.store.day.snapshot.v1 → 1 DB upsert.

4.2 KPI Exposure for the Frontend Dashboard

The exposure layer serves KPI data through two complementary patterns: synchronous pull (REST) for dashboard load and historical queries, and asynchronous push (WebSocket / SSE) for live updates. Crucially, the REST API reads exclusively from the read model database — it has no dependency on the Kafka Streams state store or on the Kafka Streams application lifecycle.

4.2.1 Architecture Overview

  • Persistence Consumer: a Kafka consumer subscribed to kpi.store.day.snapshot.v1 that performs batch idempotent upserts into the read model database and simultaneously populates Redis.
  • Read Model Database: PostgreSQL or MongoDB, optimized for point lookups and range queries by (store_id, business_date, currency). This is not the source of truth — it is a read-optimized projection of the KPI state.
  • Redis Cache: TTL-aligned with the Suppress silence window (e.g., 15 minutes). Serves the majority of REST queries at sub-millisecond latency.
  • KPI Query Service: stateless REST API reading from Redis (cache hit) or the read model DB (cache miss). No dependency on Kafka Streams.
  • Push Service: a separate Kafka consumer subscribed to kpi.store.day.current.v1 that dispatches every incoming snapshot to connected dashboard clients via WebSocket or SSE.
  • API Gateway: routes authenticated merchant requests, enforces tenant isolation and POS-level access control.

4.2.2 Persistence Consumer — Batch Upsert

Java — KpiPersistenceConsumer (batch listener)
@KafkaListener(
    topics = "kpi.store.day.snapshot.v1",
    groupId = "kpi-persistence",
    containerFactory = "batchListenerFactory"
)
public void persistBatch(List<ConsumerRecord<String, KpiSnapshot>> records,
                         Acknowledgment ack) {
    // Intra-batch dedup: keep only the most recent snapshot per composite key
    Map<String, KpiSnapshot> deduped = records.stream()
        .collect(Collectors.toMap(
            ConsumerRecord::key,
            ConsumerRecord::value,
            (a, b) -> a.getComputedAt().isAfter(b.getComputedAt()) ? a : b));

    List<KpiSnapshot> snapshots = new ArrayList<>(deduped.values());

    // Idempotent upsert: WHERE computed_at > existing (see SQL below)
    kpiRepository.upsertBatch(snapshots);

    // Populate Redis with same TTL as Suppress silence window + margin
    snapshots.forEach(snap -> redis.opsForValue().set(
        "kpi:" + snap.getStoreId() + ":" + snap.getBusinessDate() + ":" + snap.getCurrency(),
        snap, Duration.ofMinutes(15)));

    ack.acknowledge(); // manual commit AFTER successful write
}

4.2.3 Idempotent Upsert — Versioned by computedAt

SQL — PostgreSQL idempotent upsert
INSERT INTO kpi_snapshots (
    store_id, business_date, currency,
    gross_amount_cents, refund_amount_cents, net_amount_cents,
    success_count, failed_count, total_count,
    fraud_count, last_event_timestamp, computed_at
)
VALUES (:storeId, :businessDate, :currency, ...)
ON CONFLICT (store_id, business_date, currency)
DO UPDATE SET
    gross_amount_cents   = EXCLUDED.gross_amount_cents,
    refund_amount_cents  = EXCLUDED.refund_amount_cents,
    net_amount_cents     = EXCLUDED.net_amount_cents,
    success_count        = EXCLUDED.success_count,
    failed_count         = EXCLUDED.failed_count,
    total_count          = EXCLUDED.total_count,
    fraud_count          = EXCLUDED.fraud_count,
    last_event_timestamp = EXCLUDED.last_event_timestamp,
    computed_at          = EXCLUDED.computed_at
WHERE kpi_snapshots.computed_at < EXCLUDED.computed_at;
-- Guard: only overwrite if the incoming snapshot is strictly newer
-- A Kafka replay cannot regress a more recent value

4.2.4 REST Query API — DB-Only, No Kafka Streams Dependency

Java — KpiQueryController (Spring Boot)
@GetMapping("/kpi/{storeId}/date/{date}/currency/{currency}")
public KpiSnapshot getKpi(@PathVariable String storeId,
                          @PathVariable String date,
                          @PathVariable String currency) {

    String key = "kpi:" + storeId + ":" + date + ":" + currency;

    // 1. Redis cache (TTL 15 min)
    KpiSnapshot cached = redis.opsForValue().get(key);
    if (cached != null) return cached;

    // 2. Fallback: read model database
    // No Kafka Streams dependency — the REST layer is fully decoupled
    return kpiRepository.findByStoreIdAndDateAndCurrency(
            storeId, LocalDate.parse(date), currency)
        .orElse(KpiSnapshot.empty());
}

4.2.5 Cross-POS Consolidation at Query Time

Java — merchant-level consolidation endpoint
@GetMapping("/kpi/merchant/{merchantId}/date/{date}/currency/{currency}")
public KpiSnapshot getMerchantKpi(@PathVariable String merchantId,
                                   @PathVariable String date,
                                   @PathVariable String currency) {
    // All POS IDs for this merchant from merchant registry
    List<String> storeIds = merchantRegistry.getStoreIds(merchantId);

    return storeIds.stream()
        .map(sid -> getKpi(sid, date, currency))
        .reduce(KpiSnapshot.empty(), KpiSnapshot::merge);
    // merge(): sums all additive fields, recomputes derived metrics from merged sums
}

4.2.6 Real-Time Push — WebSocket / SSE

Java — KpiPushService (consumes kpi.store.day.current.v1)
@KafkaListener(topics = "kpi.store.day.current.v1", groupId = "kpi-push-service")
public void onCurrentKpi(KpiSnapshot snapshot) {
    // Push to the store's real-time channel
    ws.convertAndSend("/topic/kpi/" + snapshot.getStoreId(), snapshot);

    // Retroactive corrections: also notify the date-specific channel
    // (dashboard can update historical views without disrupting today's display)
    if (snapshot.isRetroactiveCorrection()) {
        ws.convertAndSend(
            "/topic/kpi/" + snapshot.getStoreId()
                + "/" + snapshot.getBusinessDate()
                + "/" + snapshot.getCurrency(),
            snapshot);
    }
}

4.2.7 Read Model Data Model

PRIMARY KEY: (store_id, business_date, currency)

Columns: store_id · merchant_id · business_date · currency
           gross_amount_cents · refund_amount_cents · net_amount_cents
           success_count · failed_count · total_count
           fraud_count · fraud_amount_cents
           last_event_timestamp · computed_at · is_retroactive_correction

Indexes: (merchant_id, business_date, currency) — merchant consolidation
          (store_id, business_date) — date range queries
          (computed_at) — monitoring freshness lag

Derived metrics (average basket, failure rate, net amount) are not stored — they are computed by KpiSnapshot.toSnapshot() from the stored sufficients at API response time. This ensures that any correction of stored sums immediately produces correct derived values without a schema migration.
